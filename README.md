# AWS Cloud Security Audit Lab

## Project Overview
This repository contains the documentation, methodology, and concrete evidence of a simulated Cloud Security Audit conducted within an Amazon Web Services (AWS) environment. The primary objective is to demonstrate the practical identification, architectural analysis, and remediation of common cloud misconfigurations using industry-standard auditing tools. 

By bridging the gap between technical execution and business risk management, this project reflects the core responsibilities of a Cloud Security Consultant.

## Objectives
* **Simulate** a realistic cloud infrastructure with intentional, high-risk vulnerabilities (e.g., public S3 buckets containing sensitive data).
* **Execute** automated security assessments using **Prowler**, a leading open-source security tool for AWS.
* **Analyze** findings based on the **CIS Foundations Benchmarks for AWS** and translate technical flaws into business impacts.
* **Remediate** the identified vulnerabilities by enforcing strict, compliant access controls.

## Repository Structure
* `/reports` - Contains the final executive Audit Report detailing technical findings, severity, and business impact.
* `/logs` - Archives the raw, detailed HTML and CSV output generated by the Prowler scans for forensic reference.
* `/img` - Visual evidence of the lab setup, IAM configurations, and scan executions.

## Audit Methodology

The audit follows a structured, four-phase approach aligned with standard IT-Consulting practices:

### 1. Preparation & Honeypot Setup
A deliberately vulnerable AWS environment was created to simulate a critical cloud misconfiguration. An Amazon S3 bucket was provisioned with the "Block all public access" feature explicitly disabled, mimicking a frequent cause of enterprise data breaches.

</p>

To demonstrate the tangible business impact, a file containing simulated Personally Identifiable Information (PII) and credentials was uploaded. This object was accessible via the public internet without any authentication requirements.

  <img src="img/s3-exposed-dummy-data.png" alt="Exposed Dummy Data" width="750">

### 2. Audit Setup (Least Privilege)
To conduct the automated assessment securely, a dedicated IAM user (`audit-admin`) was created. Following the fundamental security principle of **Least Privilege**, only the AWS-managed `SecurityAudit` policy was attached. This allowed the auditing tool read-only access to security configuration metadata without compromising the environment or allowing unauthorized modifications.

  <em>Fig 3: Applying the principle of least privilege for the auditing entity to ensure a non-destructive scan.</em>
</p>

### 3. Execution & Findings
The automated assessment was executed via the AWS CLI using Prowler. The scan thoroughly evaluated the account against multiple compliance frameworks, predominantly focusing on the CIS AWS Foundations Benchmark.

The scan successfully flagged the deliberately exposed S3 bucket among other default baseline misconfigurations. A detailed breakdown of the critical findings is documented in the final executive report located in `/reports/audit_report_v1.md`. 

*(Note: The full raw HTML report generated by Prowler is archived in the `/logs` directory for forensic reference).*

### 4. Remediation
Following the identification of the critical vulnerability, immediate remediation steps were taken to secure the environment. The S3 bucket's permissions were modified via the AWS Management Console to enforce strict access controls. By re-enabling the "Block all public access" safeguard, the risk of unauthorized data exposure was effectively mitigated, bringing the resource back into compliance with best practices.
